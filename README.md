# Шпаргалка по оформлению коммитам

## Шаг 1. Выберите тип изменения

В начале сообщения укажите тип, который описывает суть изменений.

- **feat** – добавление новой функциональности
- **fix** – исправление ошибки
- **docs** – изменение документации
- **style** – правки по стилю: форматирование, пробелы, запятые и т. д.
- **refactor** – рефакторинг без исправления ошибок и без добавления новой функциональности
- **perf** – оптимизация производительности
- **test** – добавление или обновление тестов
- **build** – изменения, связанные со сборкой проекта
- **ci** – настройка или изменение CI/CD
- **chore** – прочие технические изменения, например правки в `.gitignore`
- **revert** – откат предыдущего коммита

---

## Шаг 2. Укажите область изменений (`scope`, опционально)

`scope` помогает уточнить, к какой части проекта относится изменение.

Можно использовать универсальные области:

- **auth** – авторизация и аутентификация
- **ui** – пользовательский интерфейс
- **api** – серверные или клиентские API
- **core** – основной функционал приложения
- **config** – файлы конфигурации
- **deps** – обновление зависимостей
- **tests** – модульные, интеграционные или e2e-тесты
- **docs** – документация
- **db** – изменения в базе данных
- **build** – скрипты сборки или сборочные файлы

Можно использовать и свои области.  
Например, в рабочих проектах это могут быть: `form1`, `form3`, `StomachStatistics` и другие.

Пример:

```text
fix(ui): исправлен отступ у кнопки
refactor(form3): убран дублирующий код
```

## Шаг 3. Напишите краткое описание

Описание должно быть коротким и понятным.  
Обычно рекомендуют укладываться примерно в **50–70 символов**.

Можно использовать разные стили, но лучше придерживаться одного во всём проекте.

### 1. Прошедшее время

- `fix: исправлен расчет итогов`
- `refactor: убран дублирующий код`
- `style: вынесены стили таблиц`

### 2. Инфинитив

- `fix: исправить расчет итогов`
- `refactor: убрать дублирующий код`

### 3. Повелительное наклонение

- `fix: исправь расчет итогов`
- `refactor: убери дублирующий код`

В русском языке повелительная форма в коммитах часто звучит резко или неестественно.  
Поэтому в русскоязычных проектах нередко используют **инфинитив**.

Лично мне и инфинитив режет слух, поэтому я использую **прошедшее время**.

### 4. Через существительное

Есть и ещё один хороший вариант: формулировка через существительное, без глагола.

- `fix: исправление расчета итогов`
- `refactor: удаление дублирующего кода`

## Шаг 4. Дополните сообщение при необходимости

При желании после заголовка можно добавить тело коммита и описать детали:

- что именно изменено;
- зачем это сделано;
- что важно знать дополнительно.

Пример:

```text
fix(report): исправлен расчет итогов

Учтен случай пустой выборки.
Добавлена проверка деления на ноль.
```

## Шаг 5. Укажите `BREAKING CHANGE`, если нужно

Если изменение **несовместимо с предыдущей версией**, это нужно явно отметить в коммите.

Примеры:

- `refactor(sql): переименован столбец kolr в colv`  
  *(если в Python или HTML всё ещё ожидается `kolr`)*
- `refactor(sql): изменена сигнатура aged_test_form1_table1`  
  *(если вызовы функции остались старыми)*

Чтобы понять, является ли изменение `breaking change`, можно задать себе вопрос:

> Если кто-то обновит мой код, придётся ли ему что-то менять у себя?

### Варианты записи

#### 1. Через `!`

Знак `!` показывает, что изменение несовместимо с предыдущей версией:

```text
feat(api)!: изменен формат ответа отчета
```

#### 2. Через пояснение в теле коммита

```text
feat(api): изменен формат ответа отчета

BREAKING CHANGE: поле "name" переименовано в "full_name"
```

## Общий шаблон

```text
type(scope): краткое описание
```

Примеры:

```text
fix(ui): исправлен отступ у кнопки
docs(readme): добавлена инструкция по запуску
refactor(db): вынесена логика агрегации в SQL-функцию
chore(git): обновлен .gitignore
```
